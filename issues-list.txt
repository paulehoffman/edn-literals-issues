5.  ABNF Definitions 

5.1.  Overall ABNF Definition for Concise Diagnostic Notation 

Rohan: The top-level of the ABNF lists specific types of items, rather than directly including app-strings, app-sequences and elisions. While the ABNF allows app-prefixes and app-sequences indirectly through the bstr production and ellipses through the string1e production, this obfuscates the types of items they can represent. Ellipses can represent any type of item, not just text. ip'' can be a bstr or an array; dt'' is an integer; the capitalized versions are tagged.
If application extensions are buried inside the ABNF string definition, developers will assume that they are only for bstrs (like h'' and hash<<>>). Instead app-extensions and ellipsis can be directly incorporated in the grammar of `item`. [Include application extensions and ellipsis directly in top-level `item` ABNF] {{ Carsten supports }}

5.2.  ABNF Definitions for Application Extension Content 

Rohan: I have worked with ANBF for decades, and have reviewed and rewritten ABNF extensively during my career. This document presents an ABNF for most of the protocol separately from the ABNFs for prefixed strings like h’’, b64’’, ip’’, and dt’’. This requires a 2-pass parser. Most IETF protocols that use ABNF present ABNF that can be parsed as a single pass. Using the 2-pass parser means that implementors usually need to allocate dynamic memory for prefixed-strings in order to parse them later (in case they contain escape sequences). The 2-passes-by-default model has negative performance implications (which are especially relevant in CI workflows) and are more likely to run out of memory. Implementer would be free to use 2-pass parsers, but the default would be single-pass parsers. Note that there would be no change in the ABNF productions needed for ip and dt, and the change for cri is a single line. The only complexity is h and b64. I am happy to contribute a nicer-looking single-pass ABNF. [make 1-pass ABNF the default for MTI features]  {{ Carsten disagrees }}
