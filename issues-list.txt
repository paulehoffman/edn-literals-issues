2.1.  Application-Oriented Extension Literals

Rohan: I believe that the simplest mental model for extensions in CBOR is that they produce CBOR bytes and a type. This allows a unification of app-extensions with built-in features. It also makes it reasonable to remove extensibility of encoding indicators. [Proposal in EDN: App-extensions produce CBOR bytes]

Rohan: Section 2.1 implies / describes how b64`foo` is the same as b64<<"foo">> or b64<<`foo`>>. This is wrong. b64'foo # `hi`' is NOT the same as b64<<`foo # `hi``>> (which is illegal syntax).  In other words, you can't take one quoting mechanism of EDN and rewrite its syntax with another type. You could replace an app-extension not using << >> with one using << >> and the SAME string type inside (including the same number of backticks for a rawliteral). You also cannot assume that foo<< >> with a single EDN item is a type that can be expressed inside as foo'' or foo``.  For example foo<< simple(100) >> cannot be expressed any other way, even though it has a single argument. [Faulty Equivalence model of app-extension families in EDN]

Rohan: In the same way that extensions can specify if lowercase or uppercase forms are allowed, they should be able to specify the canonical form of the extension and which quoting mechanisms are allowed (if any). 5.  ABNF Definitions 

5.1.  Overall ABNF Definition for Concise Diagnostic Notation 

Rohan: The top-level of the ABNF lists specific types of items, rather than directly including app-strings, app-sequences and elisions. While the ABNF allows app-prefixes and app-sequences indirectly through the bstr production and ellipses through the string1e production, this obfuscates the types of items they can represent. Ellipses can represent any type of item, not just text. ip'' can be a bstr or an array; dt'' is an integer; the capitalized versions are tagged.
If application extensions are buried inside the ABNF string definition, developers will assume that they are only for bstrs (like h'' and hash<<>>). Instead app-extensions and ellipsis can be directly incorporated in the grammar of `item`. [Include application extensions and ellipsis directly in top-level `item` ABNF]

5.2.  ABNF Definitions for Application Extension Content 

Rohan: I have worked with ANBF for decades, and have reviewed and rewritten ABNF extensively during my career. This document presents an ABNF for most of the protocol separately from the ABNFs for prefixed strings like h’’, b64’’, ip’’, and dt’’. This requires a 2-pass parser. Most IETF protocols that use ABNF present ABNF that can be parsed as a single pass. Using the 2-pass parser means that implementors usually need to allocate dynamic memory for prefixed-strings in order to parse them later (in case they contain escape sequences). The 2-passes-by-default model has negative performance implications (which are especially relevant in CI workflows) and are more likely to run out of memory. Implementer would be free to use 2-pass parsers, but the default would be single-pass parsers. Note that there would be no change in the ABNF productions needed for ip and dt, and the change for cri is a single line. The only complexity is h and b64. I am happy to contribute a nicer-looking single-pass ABNF. [make 1-pass ABNF the default for MTI features]

5.2.1.  h: ABNF Definition of Hexadecimal representation of a byte string

Rohan: Comments inside of h'' and b64'' are there because we could end up with long sequences, and because there was no support for string concatenation. Implementations can safely use the # end-of-line comment in both h'' and b64''. There are also existing single-slash comments inside h''. However, there is no good reason to introduce support for /* */ block comments, and // end-of-line comments INSIDE h''. [Remove new comment types from inside h'' app-strings]
